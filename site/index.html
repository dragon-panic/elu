elu

a package manager that
trusts no one, including you.

github read the spec how it works

Every package manager you've used ships with arbitrary shell at install time. postinstall hooks, postinst scripts, build.rs, setup.py. That's where the supply chain attacks come from. It has been where they come from for twenty years. It was never a mystery.

elu doesn't. The operations a package can perform at install time are a closed set that elu itself implements. Anything outside that set is named run, marked dangerous, and consented to separately. Your package manager finally has a permission model. You're welcome.

two tiers. one of them is a red flag.

native ops

enforced by being elu's own code

  • chmod
  • mkdir
  • symlink
  • write
  • template
  • copy
  • move
  • delete
  • index
  • patch

Each op is a function in elu's source. The write op's entire behavior is "write these bytes to this path." There is no code path inside it that reads ~/.ssh. It cannot. On every platform. With zero kernel help. The publisher supplies arguments; elu supplies behavior.

run

the escape hatch

When a manifest uses run, elu is execve-ing a binary the package supplied. The declared reads/writes/network are disclosure, not guarantees. elu makes no promises about what the binary actually does.

Every UI surface marks run visibly. The goal is that you almost never see it. When you do, you should look.

how it works

  • closed op set, enforced by implementation. Ten declarative ops, each a function in elu's source. No shell. No plugins. Adding a new op is an elu-side code change, reviewed and shipped like any other feature. A publisher cannot smuggle in a new capability by shipping a clever manifest. → hooks.md
  • content-addressed layers, OCI-compatible. Every layer has a diff_id and a blob_id. Byte-for-byte compatible with OCI image layers. You already know how to host them, mirror them, and garbage-collect them. → layers.md
  • manifest-hash approvals with diff UX. Approvals are keyed on the manifest hash, not on name@version. An upgrade that adds a new run op, flips network to true, or widens writes re-prompts with a diff. The foo@1.0.0-is-fine-but-foo@1.0.1-eats-your-SSH-keys attack pattern stops working. → hooks.md
  • importers for apt, npm, pip. Your existing ecosystem, rewrapped as elu packages. postinst scripts do not come along for the ride. If a script maps to native ops, it's mapped. If it doesn't, it becomes a run and you see it in the approval diff. No silent execution, ever. → importers.md

what elu is not

elu is not a sandbox. Not a container. Not a VM. Not a chroot. Not a jail. Not a build system. Not a language runtime. Not a CI system. Not going to run your tests. Not going to rebuild your world from source. Not going to solve your semver constraints with a SAT solver. Not going to write your elu.toml for you. (Your coding agent will.)

elu is a package manager. That's the whole thing. If you need a sandbox, run elu inside one. If you need a build system, run your build system and hand the output to elu. If you need build-level reproducibility, that's Nix's job and Nix is good at it. elu's job starts after the bytes exist and ends when they land on disk, in the right place, with the right permissions, with consent on file.

It is boring on purpose. That is the pitch.

The PRDs are the whole design. There is no binary yet; there will be. Read them in any order.

  • README — the tour
  • manifest — the package descriptor
