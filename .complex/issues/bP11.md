## Scope (vertical slice 1 of 5 — foundation)

Wire the resolver's existing `lockfile.rs` into the CLI: implement `elu lock`, teach `install`/`stack` to read+write `./elu.lock`, and honor the global `--locked` flag.

This is the first slice because every other verb in this arc (`add`, `remove`, `update`, multi-ref `install`/`stack`) operates on the lockfile.

## Why

`crates/elu-resolver/src/lockfile.rs` already exposes `lock`, `verify`, and `update` against `Lockfile` structs, and `cli.rs:35` declares the `--locked` global flag. But `cmd/lock.rs` is a stub (dispatches to `stub::run` in `cmd/mod.rs:31`), `install.rs` and `stack.rs` never look at `./elu.lock`, and `--locked` is parsed but unused. PRD: `docs/prd/cli.md:63-71`, `docs/prd/resolver.md:126-160`.

## Files

- `crates/elu-cli/src/cmd/lock.rs` (new)
- `crates/elu-cli/src/cmd/mod.rs` (rewire `Command::Lock` from `stub::run` to `lock::run`)
- `crates/elu-cli/src/lockfile.rs` (new — small helper for read/write of `./elu.lock`)
- `crates/elu-cli/src/cmd/install.rs` (edit — load lockfile if present, write/refresh on success, honor `--locked`)
- `crates/elu-cli/src/cmd/stack.rs` (edit — load lockfile if present, honor `--locked`)
- `crates/elu-cli/tests/lock.rs` (new)

## Acceptance

- `elu lock` in a directory with `elu.toml` writes a deterministic `elu.lock`.
- `elu lock --locked` exits non-zero (code 7) when the would-be lockfile differs from disk.
- `elu install <ref>` reads `./elu.lock` if present and pins to it.
- `--locked` propagates: `elu install --locked` errors out (exit 7) if resolution would change the lockfile.
- Existing single-ref install/stack tests still pass.
